# 기존 소스 분석 및 문서 보완 워크플로우

> **Trigger:** "기존소스 분석 및 문서 보완", "Analyze Source"

이 워크플로우는 소스 코드는 존재하지만 문서가 누락되었거나 최신 상태가 아닌 프로젝트에 대해, 소스를 역공학하여 문서를 현행화하는 과정을 정의합니다.

## 1. 프로젝트 탐색 및 현황 파악

- [ ] **문서/소스 대조**: `PROJECT_DEFINITION.md` (또는 `README.md`)와 실제 소스 코드(`src`, `api` 등)의 구조적 차이를 확인합니다.
    - 주요 확인 대상: 미구현으로 표시된 로드맵 기능이 실제 코드로 구현되어 있는지 확인 (`package.json` 의존성, 파일명 등 단서 활용).
- [ ] **디렉토리 구조 분석**: `list_dir` 등을 사용하여 프로젝트의 전체 구조, 주요 진입점(`index.js`, `App.js`, `server.js` 등)을 파악합니다.

## 2. 세부 기능 구현 확인

- [ ] **핵심 로직 확인**: 의심되는 기능(예: 로그인, 저장 등)의 구현 코드를 `view_file`로 직접 확인합니다.
    - 예: '로그인' 기능이 있다면 `auth` 관련 파일이나 `supabase`, `firebase` 초기화 코드를 확인.
- [ ] **API 엔드포인트 확인**: 백엔드/서버리스 함수 디렉토리(`api/`, `routes/`)를 스캔하여 실제 제공되는 API 목록을 파악합니다.

## 3. 문서 최신화 (Documentation Update)

- [ ] **프로젝트 정의서 업데이트**: 확인된 구현 내용을 바탕으로 `PROJECT_DEFINITION.md`의 '현재 구현된 기능' 및 '로드맵' 상태를 `[x]`로 업데이트합니다.
    - **버전/날짜 갱신**: 문서의 버전과 수정일을 오늘 날짜로 갱신합니다.
- [ ] **저널링 (선택)**: `PROJECT_JOURNAL.md`에 "기존 구현분 발견 및 문서화" 또는 역추적된 개발 이력을 기록합니다.

## 4. 누락된 문서 생성

- [ ] **없는 문서 보완**: 프로젝트에 필수적인 문서(`init.md` 등)가 없다면 소스 코드의 설정 파일(`package.json`, `vercel.json` 등)을 근거로 작성합니다.
